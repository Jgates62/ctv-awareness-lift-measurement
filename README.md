# CTV Brand Lift: Scalable ELT Pipeline

## Executive Summary

This project provides a foundational data architecture for measuring **Brand Lift** in a programmatic Connected TV (CTV) environment. It simulates the end-to-end lifecycle of ad exposure and consumer sentiment measurement - from raw bidstream logs to an analysis-ready attribution dataset.

The pipeline is built to bring clarity to awareness advertising effectiveness by merging high-volume telemetry with nuanced consumer survey data.

## Project Architecture

To ensure scalability, this project utilizes an **ELT (Extract, Load, Transform)** paradigm. Unlike traditional ETL scripts that process data in local memory, this architecture pushes all heavy computation (joins, window functions, and deduplication) directly into BigQuery.

### 1. Data Generation (`data_generator.py`)

This script simulates a multi-table environment representing a realistic ads ecosystem:

- **Impression Logs** - Mimics programmatic bidstream data with funnel metrics like `ad_started` and `ad_completed`.
- **Survey Responses** - Generates sentiment data for *Exposed* vs. *Control* groups, introducing simulated lift for the exposed group to validate measurement models.
- **Study Metadata** - Maps specific measurement studies to arrays of Campaign IDs, supporting the one-to-many relationship between high-level brand goals and specific ad tactics.
- **Ingestion to BigQuery** - Uses the official `google-cloud-bigquery` client with `WRITE_TRUNCATE` for idempotent, production-consistent data loading.

### 2. The ELT Engine (`ELT.py`)

The transformation layer is designed for data integrity and performance:

- **Warehouse-Native Execution** - Transformations are executed as BigQuery SQL queries to minimize data egress and leverage Google's distributed compute.
- **Deduplication** - Uses `ROW_NUMBER() OVER(PARTITION BY IP ...)` to ensure that if a user takes a survey multiple times, only their first entry is counted.
- **Temporal Attribution** - Implements a strict temporal join (`response_dt >= first_imp_dt`) to ensure users are only categorized as *Exposed* if their ad impression occurred before they took the survey.
- **Parameterized Queries** - Protects against SQL injection and optimizes query plan caching.

## Key Design Principles

This project demonstrates a set of core competencies relevant to modern ad measurement and data engineering:

| Capability | Implementation |
|---|---|
| **Foundational Datasets** | Designing metrics (Positive Response Rate, Ad Exposure) that drive business decisions |
| **0 to 1 Execution** | Building a functional prototype from scratch, including data generation and processing |
| **Scalability** | Moving away from Pandas-based joins toward warehouse-native SQL to handle large-scale data |
| **Domain Expertise** | Understanding Ad-Tech nuances like campaign mapping, attribution windows, and experimental group design |

## Getting Started

**1. Configure Environment**

Set your Google Application Credentials, Project ID, and Dataset ID in a `.env` file:

**Please don't share your credentials publicly! You can use `.gitignore` to exclude `.env` from your GitHub repo.**

```env
GOOGLE_APPLICATION_CREDENTIALS=your-gcp-credentials.json
PROJECT_ID=your-gcp-project-id
DATASET_ID=your-bigquery-dataset-id
```

**2. Generate Data**

Run the data generator to provision the foundational tables:

```bash
python data_generator.py
```

**3. Execute Transformation**

Run the ELT engine to generate the final analysis table in BigQuery:

```bash
python ELT.py
```
